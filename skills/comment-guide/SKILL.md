# 代码注释指南

## 这个 skill 的使用节奏

注释介入应该是**信号驱动**的，不是节奏驱动的。

- ❌ 不是"每写完一个函数都跑一遍自检"——那是仪式，会拖慢正常工作
- ❌ 不是"每读一段代码都顺手审查注释"——那会偏离用户当前的真正意图
- ✅ 是"代码里出现了值得解释的特征，就停一下决定怎么做"
- ✅ 是"用户明确要求审查注释时，按方法论系统性扫"

写普通业务逻辑、改变量名、跑测试、回答"这函数干嘛的"——**这些时候不要跳出来管注释**。代码自己说得清的事，让代码说。

---

## 写代码时：信号清单

只在以下信号出现时停一下，问"这里要不要写/改注释？"。其他时候继续干活。

### 强信号（看到就该处理，不处理是欠债）

- **你做出了反直觉的决定**：明明可以更短、更直接、更"正常"地写，却偏偏选了一条绕路。绕路的理由不留下来，下一个人会把它"修"回错的版本
- **你写了 workaround**：绕过 bug、规范缺陷、上游限制、性能坑。**为什么绕、绕了什么、什么时候可以拆掉**——三件事都得说
- **你新增或修改了公共接口**：被外部调用者依赖的函数、类型、模块。前置条件、返回语义、错误情况、副作用——签名说不清的部分必须说出来
- **你引入了跨层契约**：编译器和类型系统检查不到的不变量（"调用方持有锁"、"输入已经过验证"、"此值生命周期由 X 管理"）
- **你的修改紧邻一段已有注释**：那段注释现在是否仍然准确？哪怕你这次没改注释行，也要扫一眼——很多过时注释就是这样产生的
- **你写下了一个有意识的偏离/限制**：不完整实现、未支持的边界情况、未来要扩展的位置。当下的行为、为什么这样、踩进会怎样——说清楚

### 弱信号（视情况处理，宁缺毋滥）

- **较长或较深嵌套的逻辑**：入口处一句"整体在干什么"的概述，让读者不必先读完才知道在读什么
- **基于位、字节、内存布局的低级技巧**：解释**语义**，不是机械步骤
- **正则、模式匹配、状态机**：当规则本身有非平凡含义、而代码只表达机制时

### 反信号（出现就**不要**写注释）

- 良好命名已经传达的意图（`is_valid_user(...)` 旁边不需要"检查用户是否有效"）
- 显而易见的控制流（普通 `if`/`for`/`while` 不要"如果...则..."的解说）
- 简单转发与委托（一行 `return inner.foo(x)` 旁边的注释多半是噪音）
- 重述代码字面（`i += 1` 旁边写"自增 i"）

## 在已识别的信号上做判断：三问

只有上面的信号**已经触发**之后，才用这三问决定具体怎么动笔（或决定不动）：

1. **下一个维护者读到这里，会不会停下来想"为什么这么写"？**
   会 → 那个"为什么"就是这条注释要回答的内容
   不会 → 不要写

2. **这块代码周围的现有注释，是否仍然和代码一致？**
   不一致 → 以代码为准，注释要么改、要么删。**保留错的注释比没注释更坏**

3. **这段写法是否在违反某种"看起来更自然"的写法？**
   是 → 解释为什么不能那样写（规范、性能、兼容、踩过的坑）
   否 → 不需要"自我辩护"

三问跑完即得结论，不要扩张到无关位置。

## 何时该闭嘴

这个 skill 最重要的判断之一是**不介入**。以下场景明确不要主动谈论注释：

- 用户问的是行为/用法/调试，不是文档（"这函数怎么用"——直接答，不要顺手审查注释）
- 改动是机械重命名、格式化、移动文件——注释只要跟随代码移动即可
- 写的是测试代码——除非测试本身在断言反直觉的不变量，否则测试名+断言已经够说明
- 改动很小且与注释无交集——不要趁机"顺便美化"周围注释，那会污染 diff
- 用户没有问注释、代码也没有触发信号——保持沉默

## 注释审查（用户明确要求时）

写注释是信号驱动的。**审查注释是用户请求驱动的**——不要主动跑全文扫描，那是越权。

当用户说"看看注释"、"审查注释"、"注释有没有过时"、"review comments"等等时，按下面方法论系统性扫一遍。

### 扫描重点（按价值密度排序）

1. **`TODO` / `FIXME` / `XXX` / `HACK` / `DEPRECATED` 标记**：逐条核对当前代码是否仍处于注释描述的状态。已做完的删；改方向的更新；仍有效的补足上下文（**为什么、什么场景下出问题、可能怎么解**）
2. **限制性表述**："当前仅支持..."、"暂未实现..."、"不符合..."、"未来需要..."——验证今天是否仍然成立。**已修复的最容易留下陈旧注释**
3. **指向外部规范、issue、文档的引用**：链接是否还有效、章节引用是否仍准确、规范是否升级
4. **描述行为的注释 vs 实际代码**：代码改过、注释忘了改是最常见的过时形式。任何"做了什么"型注释都要拿代码核对
5. **重述型噪音**：上一节反信号里的那种——它们没"过时"，但本不该存在，遇到顺手清掉

### 报告方式

按发现讲清楚就行，不需要固定模板。让用户能据此决定**改还是不改**就够了。一般包含：

- 在哪（文件 + 行号或符号）
- 是什么问题（陈旧 / 噪音 / 缺失 / 误导）
- 用什么证据判断（代码现状 vs 注释描述的差异）
- 建议（改写 / 删除 / 补上 / 保留）

**发现量大时，先报关键的几条，问用户优先关心哪类，再继续下沉。**一次性甩 200 条会淹没决策。

## 写注释的方式（语言中立）

具体语法（行注释 / 块注释 / 文档注释 / docstring）跟随上下文：

- 看周围已有注释怎么写——风格、人称、详略，保持一致
- 看项目的 `AGENTS.md` / `CLAUDE.md` / `.editorconfig` / 风格文档——它们如果规定了注释语言（中文/英文）或文档注释格式，**严格遵循**
- 文档注释要符合该语言生态工具链能解析的格式
- 没有任何提示时，使用与该文件已有非注释文本一致的自然语言

内容上：

- 解释 **why**，不重复 **what**
- 一段代码只需要一个解释入口，不要在每行都注释
- 长度跟随被解释逻辑的复杂度——一句话能讲清就不用三段
- 用陈述句，不用废话起手（"这个函数是用来..."、"这里我们..."这类开头通常可以删掉）

## 如何写一条好注释

知道了"该不该写"和"写在哪"，下面是动笔时怎么写。

### 写之前的三问

写出第一个字之前，问自己：

1. **六个月后的我读到这里会停下来问什么？** —— 那个问题就是这条注释要回答的内容。如果想不出会被问什么，多半不该写
2. **一个新人不看这段代码的历史和上下文，凭这条注释能否知道"为什么这样写"？** —— 不能则说明你只写了 what，没写 why
3. **这条注释会不会随着代码继续演化而变成谎言？** —— 会则现在就加锚点（issue/PR 编号、规范章节、commit hash），或者改成不随时间漂移的措辞

第三问最容易被忽视，但和"过时注释"那一节直接闭环：**写的时候多花一秒锚定，后面就少一条要清理的废注释**。

### 内容骨架（按需取用，不是模板）

一条注释里可能需要的维度，从必要到可选：

- **决策**：为什么这么做。**这是核心**，几乎所有有价值的注释至少有这一层
- **背景**：如果"在解决什么问题"不显然，先交代一句
- **替代**：仅当存在一种"看起来明显更自然但实际错"的替代写法时，说明为什么不能那样写——防止下一个人把它"修"成错的
- **限制 / 后果**：仅当当前实现有未覆盖的边界、已知缺陷、要传递给调用者的注意事项时

不是每条注释都要四样齐全。判断标准：

- 简单的局部决策 → 只要"决策"一句话
- 公共 API 文档 → 通常"背景 + 决策 + 限制"
- workaround / 反直觉决定 → 往往四样齐全（这是它复杂的原因）

### 写作习惯

- **主动语态**："调用方持有 `lock`"，不写"`lock` 应被持有"
- **用具体名词代替模糊代词**："this"、"it"、"这里"、"那个"在注释里几乎都该换成具体名字。代码改动后，模糊代词最容易指错
- **用永久锚点，不用相对时间**：避开"最近"、"现在"、"暂时"、"目前"、"之后"——这些词从写下那刻起就在过期。换成 issue 号、PR 号、规范章节号、版本号、commit hash
- **TODO 必须包含三件事**：
  - 为什么现在没做（缺什么前置条件、哪个决策悬而未决）
  - 什么条件下应该做（具体触发，不是 wishful "以后"）
  - 大致方向（不必给完整方案，让接手的人不必从零思考）
  - 写不出这三件事的 TODO 就直接删掉，它没有信息量
- **文档注释从调用者视角**：用户怎么用、需要什么前提、返回什么、什么情况会出错。不要泄漏内部实现细节
- **行内注释从作者视角**：为什么在这一行做这个决策。不要重复讲实现做了什么——代码已经讲了

### 写完后的自检

落笔之后再看一遍，三件事：

1. **试着删掉**：删了下一个人会损失什么具体信息？说不出具体损失就真的删掉
2. **把代码遮住，只看注释**：能不能独立立住？立得住说明你写到了 why；立不住说明在重述代码
3. **演练一年后**：这条注释里有哪个词、哪个事实、哪个引用，一年后可能变成谎言？提前锚定它

这三步通常只需要十几秒，但能筛掉绝大部分没有寿命的注释。

## 反例对照

代码用语言中立的伪写法。要点是**模式**，不是语法。

### ❌ 噪音注释

```
// 自增计数器
counter += 1

// 如果用户存在则返回
if user is not None:
    return user

// TODO: fix
parse(input)

// 这是一个工厂函数，用来创建 Foo
def make_foo(...): ...
```

每条都在重述代码或留下无意义痕迹。删掉，代码本身更清晰。

### ✅ 有价值的注释

```
// 必须先递增再读取：递减场景下并发读者会看到中间状态，
// 这个顺序避免了 issue #1234 中描述的竞态。
counter += 1
value = counter

// 规范要求 Symbol 键不参与 for-in，所以这里跳过非 String/Number 键
for key in own_keys:
    if not is_enumerable_string_or_number(key): continue
    ...

// 暂未实现 [[PrivateElements]] 真实槽位，使用 "#name" 字符串键近似。
// 后果：跨类同名私有字段会冲突，外部代码可通过该字符串键读取。
// 完整实现需要在对象布局里增加私有槽段。
def get_private_field(obj, name): ...
```

每条都解释了**代码无法靠自己说清的东西**：竞态原因、规范条款、当前实现的缺陷与影响。

### ❌ 演化型陈旧（今天写下时就埋好了过期种子）

```
// 暂时用字符串作为私有字段键
def get_private_field(...): ...

// 最近 spec 改了这个行为
if is_old_behavior: ...

// TODO: 性能优化
def hot_path(...): ...

// 这里 hack 一下，因为 X 库有 bug
workaround()
```

"暂时"、"最近"、"性能优化"、"X 库的 bug"——每一条都没说清**到底是什么、什么时候可以拆**，三年后没人知道还要不要保留。

### ✅ 自带锚点的同款注释

```
// 使用 "#name" 字符串键近似 [[PrivateElements]]，issue #123 跟踪真实槽位实现
def get_private_field(...): ...

// ES2022 spec §10.4.5 修订后行为，旧分支保留是为了 v1 配置兼容（v2 起可删）
if is_old_behavior: ...

// TODO: 当 collisions > 100 时改用 robin-hood 探测；常规路径已优化，
// 此处 O(n²) 仅在 hash collision 极端情况下命中
def hot_path(...): ...

// 绕过 X 库 0.7 的回调泄漏（upstream X/issue/123，0.8 已修，升级后删此分支）
workaround()
```

每一条都给出了**判断它是否仍然有效的依据**——issue 号、规范条款、版本号、可观测的触发条件。

## 收尾时的诚实

如果你在一次任务里改了代码：

- 把这次改动**直接相关**的注释顺手维护好——这是改动的一部分，不是额外工作
- 不要去"顺便美化"无关注释——污染 diff、放大 review 成本
- 在改动过程中如果偶遇陈旧注释但不打算这次修，**说出来**让用户决定要不要单独处理；不要假装没看见，也不要擅自扩大改动范围

注释的可信度是项目的公共资源。每一条留下的陈旧注释都在消耗这份信用。
